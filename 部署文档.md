# SmartScribe 服务器部署文档

若你希望使用 **Vercel（前端）+ Render（后端 API）** 的云端部署，请参阅 [`部署文档-Vercel-Render.md`](部署文档-Vercel-Render.md)。下文为自建 Linux 服务器方案，两套文档并行保留。

本文档默认采用单台 Linux 服务器部署：

- 前端：Vite 构建产物，由 Nginx 托管
- 后端：Node.js + Express，由 PM2 守护
- 数据库：MySQL
- 访问方式：浏览器访问同域名站点，`/api` 由 Nginx 反向代理到后端

**已上线服务器只需更新代码时**，直接阅读 [§9 更新线上代码](#9-更新线上代码)，无需重复完整安装流程。

## 1. 部署前准备

服务器建议准备以下环境：

- Node.js 18 及以上
- npm 9 及以上
- MySQL 8.x
- Nginx
- PM2

安装 PM2：

```bash
npm install -g pm2
```

## 2. 拉取代码并安装依赖

```bash
git clone git@github.com:lls174/SmartScribe.git SmartScribe
cd SmartScribe

cd backend
npm install

cd ../frontend
npm install
```

说明：

- 后端 `npm install` 需包含 `devDependencies`（含 `tsx`、`typescript`），否则数据库迁移脚本无法执行。不要使用 `npm install --omit=dev`。
- 前端安装依赖时会自动执行 `postinstall`，将 Live2D 运行库复制到 `public/`。如果这一步失败，桌宠相关资源可能无法正常加载。

## 3. 配置环境变量

### 后端环境变量

复制示例文件并修改：

```bash
cd backend
cp .env.example .env
```

至少需要确认以下变量：

- `NODE_ENV=production`
- `PORT=3001`
- `DB_HOST`
- `DB_PORT`
- `DB_USER`
- `DB_PASSWORD`
- `DB_NAME`
- `JWT_SECRET`
- `AI_KEY_ENCRYPTION_SECRET`（用户 AI 密钥加密存储，必填）
- `CORS_ORIGINS`
- `TRUST_PROXY=true`
- `ADMIN_BOOTSTRAP_KEY`
- `GLM_AI_KEY` 等 AI 平台密钥（可选；未配置时用户可在设置页保存个人密钥）

示例：

```env
PORT=3001
NODE_ENV=production
DB_HOST=127.0.0.1
DB_PORT=3306
DB_USER=smart_scribe
DB_PASSWORD=your_password
DB_NAME=smart_scribe
JWT_SECRET=replace_with_a_long_random_string
AI_KEY_ENCRYPTION_SECRET=replace_with_another_long_random_string
ADMIN_BOOTSTRAP_KEY=replace_with_a_long_random_string
CORS_ORIGINS=https://your-domain.com
TRUST_PROXY=true
GLM_AI_KEY=
```

### 前端环境变量

复制示例文件并修改：

```bash
cd ../frontend
cp .env.example .env.production
```

默认推荐保持同源部署：

```env
VITE_APP_BASE_URL=/
VITE_API_BASE_URL=/api
```

说明：

- `VITE_APP_BASE_URL=/` 表示前端部署在站点根路径
- `VITE_API_BASE_URL=/api` 表示前端通过同域 `/api` 访问后端
- 如果你需要把前端挂在子路径，例如 `/smartscribe/`，请同时把 `VITE_APP_BASE_URL` 改为 `/smartscribe/`

## 4. 初始化数据库

先在 MySQL 中创建数据库和账号，然后于 `backend` 目录执行。

### 4.0 脚本说明

| 命令 | 适用场景 | 说明 |
| ---- | -------- | ---- |
| `npm run db:init` | **全新空库** | 创建全部业务表（含 `novel_versions`、`ai_credentials` 等） |
| `npm run db:migrate:rbac-usage` | 旧库升级 | 补齐 `users` RBAC 字段，创建 `ai_request_logs` |
| `npm run db:migrate:novel-memory` | 旧库升级 | 创建人物卡、内容设定表 |
| `npm run db:migrate:novel-version` | 旧库升级 | 仅创建 `novel_versions`（版本快照功能） |
| `npm run db:migrate:ai-credential` | 旧库升级 | 创建 `ai_credentials`（用户 AI 密钥加密存储） |

说明：

- **已有业务数据的库**：按需执行上表中的 `db:migrate:*`，**不要**对非空库执行 `db:init`。
- 迁移脚本依赖 `tsx`（位于 `devDependencies`）。服务器请执行完整 `npm install`，不要使用 `npm install --omit=dev`，否则 `db:init` 及部分迁移会失败。
- 所有迁移脚本尽量幂等，可重复执行。

### 4.1 全新部署（空库）

```bash
cd backend
npm run db:init
```

成功后会输出「数据库初始化成功！」。

### 4.2 从旧版本升级（已有数据）

在 `backend` 目录按需执行（通常只需补跑缺失项）：

```bash
cd backend

# RBAC、AI 用量日志（较早期版本）
npm run db:migrate:rbac-usage

# 人物卡、内容设定
npm run db:migrate:novel-memory

# 小说版本快照（访问 /api/novel/:id/versions 报「版本功能未初始化」时执行）
npm run db:migrate:novel-version

# 用户 AI 密钥加密存储
npm run db:migrate:ai-credential
```

### 4.3 初始化管理员（admin）

在后端 `.env` 配置好 `ADMIN_BOOTSTRAP_KEY` 后，执行一次初始化请求（只用于创建/提升首个管理员账号）：

```bash
curl -X POST "https://your-domain.com/api/admin/bootstrap" \
  -H "Content-Type: application/json" \
  -H "x-admin-bootstrap-key: <你的ADMIN_BOOTSTRAP_KEY>" \
  -d '{"username":"admin","password":"admin123456"}'
```

## 5. 构建前端

```bash
cd frontend
npm run build
```

构建产物位于：

```bash
frontend/dist
```

## 6. 构建并启动后端

后端为 TypeScript 项目，生产环境需先编译再启动。

```bash
cd backend
npm run build
```

推荐使用 PM2 通过 `npm start` 启动（入口为 `dist/backend/src/app.js`）：

```bash
pm2 start npm --name smartscribe-backend -- start
pm2 save
pm2 startup
```

或直接使用 node：

```bash
pm2 start dist/backend/src/app.js --name smartscribe-backend
pm2 save
pm2 startup
```

常用命令：

```bash
pm2 status
pm2 logs smartscribe-backend
pm2 restart smartscribe-backend
pm2 stop smartscribe-backend
```

## 7. 配置 Nginx

示例配置如下，请把域名和目录替换成你的实际路径：

```nginx
server {
    listen 80;
    server_name your-domain.com;

    root /var/www/SmartScribe/frontend/dist;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:3001/api/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 300s;
        proxy_send_timeout 300s;
        proxy_buffering off;
    }
}
```

说明：

- `try_files $uri $uri/ /index.html;` 用于支持 React Router 刷新不 404
- `proxy_buffering off;` 对 AI 流式输出更友好
- 如果启用 HTTPS，请在证书配置完成后把 `CORS_ORIGINS` 配置为实际的 `https://` 域名

## 8. 上线检查

后端健康检查：

```bash
curl http://127.0.0.1:3001/api/health
```

正常返回示例：

```json
{
  "status": "ok",
  "database": {
    "connected": true,
    "lastCheckedAt": "2026-04-18T00:00:00.000Z",
    "lastError": null
  },
  "uptime": 12.34
}
```

前端检查项：

1. 打开首页不报错
2. 刷新 `/login`、`/creation`、`/novel/1` 等路由不出现 404
3. 登录、创建小说、请求 AI 接口时浏览器网络面板中的请求地址为同域 `/api/...`

## 9. 更新线上代码

项目已首次部署完成后，日常发版按本节操作即可。以下命令默认在项目根目录 `/var/www/SmartScribe`（或你实际 clone 的路径）执行。

### 9.1 标准更新流程（推荐）

```bash
# 1. 进入项目目录
cd /var/www/SmartScribe

# 2. 拉取最新代码（主分支示例，按你实际分支名替换）
git fetch origin
git pull origin main

# 3. 更新后端依赖、编译并重启
cd backend
npm install
npm run build
pm2 restart smartscribe-backend

# 4. 更新前端依赖并重新构建
cd ../frontend
npm install
npm run build

# 5. 验证（见 9.4）
```

说明：

- 前端构建产物会写入 `frontend/dist`，Nginx 的 `root` 已指向该目录时**无需改 Nginx 配置**，刷新浏览器即可看到新版本（若被缓存，可强制刷新或清 CDN）。
- 后端由 PM2 守护，`pm2 restart` 会加载新代码；若改了 `.env`，同样需要执行一次 `pm2 restart smartscribe-backend`。
- 若本次更新**没有**改 `package.json` / `package-lock.json`，可省略对应目录的 `npm install`，但发版时执行一次更稳妥。

### 9.2 何时需要数据库迁移

仅当版本说明要求、或接口提示缺少数据表时，在**重启后端之前**于 `backend` 目录执行（可重复执行，脚本尽量幂等）：

```bash
cd /var/www/SmartScribe/backend

# 按需执行，参见 §4.0 脚本说明
npm run db:migrate:rbac-usage
npm run db:migrate:novel-memory
npm run db:migrate:novel-version
npm run db:migrate:ai-credential
```

常见对应关系：

| 现象 | 建议执行 |
| ---- | -------- |
| 版本列表/保存版本报「版本功能未初始化」 | `db:migrate:novel-version` |
| 设置页保存 AI 密钥失败或报缺表 | `db:migrate:ai-credential` |
| 人物卡、内容设定相关报错 | `db:migrate:novel-memory` |
| 管理员/用量统计相关缺字段 | `db:migrate:rbac-usage` |

全新空库仍用 `npm run db:init`，不要与迁移混用在同一次「空库」场景。

### 9.3 环境变量或 Nginx 有变更时

| 变更类型 | 操作 |
| -------- | ---- |
| 后端 `.env` | 编辑 `backend/.env` 后执行 `pm2 restart smartscribe-backend` |
| 前端 `.env.production` | 修改后必须重新 `cd frontend && npm run build` |
| Nginx 配置 | `sudo nginx -t` 通过后 `sudo systemctl reload nginx` |

### 9.4 更新后检查

```bash
# 后端健康
curl http://127.0.0.1:3001/api/health

# PM2 状态
pm2 status
pm2 logs smartscribe-backend --lines 50
```

浏览器侧：打开站点首页、登录、走一遍核心功能；若前端无变化，尝试 Ctrl+F5 或无痕窗口排除缓存。

### 9.5 回滚（发布异常时）

```bash
cd /var/www/SmartScribe
git log --oneline -5          # 查看最近提交
git checkout <上一个稳定提交的 hash>

cd backend
npm install
npm run build
pm2 restart smartscribe-backend
cd ../frontend && npm install && npm run build
```

回滚后同样执行 9.4 的健康检查。数据库迁移一般**不要**随意回滚表结构；若新版本已执行迁移且需回退代码，请先确认旧版本是否兼容当前库表。

### 9.6 一键更新脚本（可选）

可将以下内容保存为服务器上的 `scripts/deploy-update.sh`（路径自定），赋权后使用：`chmod +x scripts/deploy-update.sh && ./scripts/deploy-update.sh`。

```bash
#!/usr/bin/env bash
set -euo pipefail

PROJECT_DIR="${PROJECT_DIR:-/var/www/SmartScribe}"
BRANCH="${BRANCH:-main}"

cd "$PROJECT_DIR"
git fetch origin
git pull origin "$BRANCH"

cd backend
npm install
npm run build
pm2 restart smartscribe-backend

cd ../frontend
npm install
npm run build

echo "更新完成。请执行: curl http://127.0.0.1:3001/api/health"
```

需要迁移时，在脚本中 `pm2 restart` 之前自行增加 `npm run db:migrate:...` 行，或发版前手动执行一次。

## 10. 常见问题

### 1. 浏览器提示跨域错误

检查后端 `.env` 中的 `CORS_ORIGINS` 是否包含前端实际访问地址，例如：

```env
CORS_ORIGINS=https://your-domain.com
```

### 2. 页面刷新后 404

说明 Nginx 没有正确配置：

```nginx
try_files $uri $uri/ /index.html;
```

### 3. 登录或写入接口返回 `CSRF 验证失败`

优先检查：

- 前端和后端是否通过同域访问
- Nginx 是否正确转发 `/api`
- 是否使用了 HTTPS
- 浏览器是否能正常拿到 `/api/csrf-token`

### 4. 健康检查返回 `degraded`

说明后端进程已启动，但数据库不可用。请检查：

- MySQL 是否启动
- `.env` 中的 `DB_*` 配置是否正确
- 数据库用户是否有目标库权限

### 5. AI 接口长时间无响应

请检查：

- Nginx 反向代理是否关闭了缓冲
- 服务器是否能访问对应 AI 平台
- 是否已在设置页保存个人 AI 密钥，或在后端 `.env` 中配置 `GLM_AI_KEY` 等默认密钥

### 6. `npm run db:init` 报找不到模块

若报错 `Cannot find module '.../scripts/initDb.ts'`，说明代码版本过旧，请先 `git pull` 拉取最新代码后再执行。确保 `backend/scripts/initDb.ts` 存在。

### 7. 版本功能报「未初始化」

接口返回「版本功能未初始化，请先在后端执行 npm run db:init」时，**已有数据的库**应执行：

```bash
cd backend
npm run db:migrate:novel-version
```

不要对已有业务数据的库执行 `db:init`。
